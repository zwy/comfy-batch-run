# 批量生成提示词功能优化设计（多样性优先）

## 背景与目标

当前 `gen-prompts` 的实现是单次请求后按行切分，存在以下风险：
- 当模型输出重复内容时，多样性不足；
- 可能出现实际可用条数小于 `count`；
- 输出格式可能混入编号或解释文本。

本次优化目标（保持 CLI 不变）：
1. 在不修改 `python main.py gen-prompts ...` 用法的前提下，显著降低重复率；
2. 尽量稳定达到目标数量 `count`；
3. 保持输出文件为“每行一条纯提示词”。
4. 默认允许生成 NSFW/成人裸露类提示词，不在本地做内容过滤。

## 方案选择

选用 **方案 B：分批生成 + 聚合去重**（用户确认）。

不采用的方案：
- 单次生成 + 后处理补齐：改动小，但在高重复场景下补齐效果不稳定；
- 仅增强模板约束：实现简单，但对不同模型的稳定性不够。

## 架构设计

仅调整 `PromptGenerator.generate()` 内部流程，不新增 CLI 参数。

新增内部步骤（可在 `prompt_tool/prompt_generator.py` 内以私有方法实现）：
1. **批次规划器**：根据 `count` 计算每批请求数量与最大批次数；
2. **输出清洗器**：去编号、去解释前缀、去空行，提取候选提示词；
3. **规范化去重器**：对候选词做规范化键（大小写/空白/常见分隔符）后去重；
4. **补齐调度器**：若不足 `count`，继续请求下一批并注入“避免重复”约束；
5. **终止策略**：达到 `count` 或超过重试上限后结束。

## 数据流设计

1. 读取示例提示词，构建基础用户消息；
2. 发起第 1 批请求，解析并清洗候选；
3. 对候选做规范化去重后写入聚合池；
4. 若聚合池数量不足：
   - 在下一批提示中加入“不要重复已有结果”的约束；
   - 附带少量已生成样本摘要，要求主题/构图变化；
   - 轻微调整温度（如 0.85~1.0）提升探索度；
5. 循环直到满足终止条件；
6. 返回前 `count` 条并写入输出文件。

### NSFW 生成约束

- 系统提示词与用户提示词中显式声明：可生成成人/裸露类内容提示词；
- 本地不增加任何 NSFW 过滤或替换规则，避免把目标内容“清洗掉”；
- 若模型服务端策略拒绝返回（安全拦截），保留原始错误并明确提示“由模型服务端拦截”。

## 错误处理

- API 请求失败：按批次重试（有限次数），失败后抛出明确异常；
- 响应为空或不可解析：计入失败并继续下一批；
- 响应出现拒绝语（例如“无法生成成人内容”）：按“服务端拦截/拒绝”路径处理并重试下一批；
- 达到上限仍不足 `count`：返回已生成结果，同时给出缺口说明（不吞错）。

## 测评设计

使用可复现的 mock 测试验证，不依赖真实 LLM：

1. **数量达标测试**  
   输入高重复 mock 输出，验证补齐逻辑可尽量达到 `count`。

2. **去重有效性测试**  
   混入大小写差异、编号前缀、空白差异，验证最终集合去重正确。

3. **格式纯净测试**  
   验证输出不包含编号/解释文本，每行一条可直接用于生图。

4. **失败路径测试**  
   模拟 API 连续失败，验证异常信息明确且不静默失败。

## 兼容性与影响

- **CLI 兼容**：命令与参数保持不变；
- **行为变化**：生成过程可能多次请求 LLM，换取更高多样性与稳定条数；
- **成本影响**：可能增加 token 消耗（与重复率相关）。

## 实施边界

本次只覆盖 `gen-prompts` 生成链路，不改动 `batch-run` 生图链路与 ComfyUI 客户端。
